# 🛡️ Киберполигон Smart Wiki — Проектное руководство для команды

## 1. Что мы строим
Мы создаём изолированный испытательный стенд , эмулирующий защищенную корпоративную базу знаний (Wiki) со встроенным ИИ-ассистентом. Наша цель — реализовать концепцию **Secure by Design**, выстроив эшелонированную защиту, и на практике исследовать уязвимости систем, которые доверяют выводу LLM и передают его дальше без проверок.

Стенд физически разделен на две изолированные виртуальные машины:

* **Ubuntu Host (Сервер):** Запускает ядро защиты, нейросети и базы данных.

* **Kali Linux (Атакующий):** Станция для проведения атак (Prompt Injection, XSS, RCE).

## 2. Архитектура и взаимодействие компонентов

Система строго разделяет потоки данных (запросы пользователей) и телеметрию (мониторинг). 

### 2.1. Surface Layer (Внешний контур)
Определяет роли и уровни доступа к системе:

* **User (Сотрудник):** Взаимодействует только со Streamlit UI по стандартному HTTP. Не имеет прямого доступа к API или БД.

* **Admin (Аудитор ИБ):** Имеет защищенный доступ только к панели мониторинга (Grafana) и логам (Langfuse) для анализа инцидентов.

* **Kali Linux (Атакующий):** Пытается пробить систему через веб-интерфейс или прямыми запросами к API.

### 2.2. Interface Layer (Пользовательский интерфейс)
**Streamlit (Wiki UI):** Изолированная среда отображения. Рендерит ответы ИИ. Все запросы отсюда уходят исключительно в слой API. 

### 2.3. API Layer (Ядро защиты - FastAPI)
Центральный мозг системы, связующий все узлы.
* **GuardIn (Входной фильтр):** Единая точка входа. Проверяет текст запроса на паттерны инъекций. Если найдена угроза — блокирует запрос (403).
* **Router (Диспетчер):** Распределяет валидные запросы по режимам (Chat, RAG, Agent).
* **GuardOut (Выходной фильтр):** Строгий цензор. Проверяет ответ LLM на наличие `<script>` тегов, команд удаления таблиц или утечек конфиденциальных данных.

### 2.4. AI / LLM Layer (ИИ и Логика)
Работает в закрытом внутреннем контуре.
* **Ollama:** Сервер локальных моделей. Не имеет доступа в интернет.
* **LangGraph / LangChain:** Реализует логику агентов (цикл Think/Act) и пайплайны RAG. 

### 2.5. Data & Tools Layer (Хранилища)
* **MCP Server:** Сервер инструментов для Агента. Содержит намеренно уязвимые функции (чтение файлов, SQL) для демонстрации RCE.
* **ChromaDB:** Векторная база для поиска по корпоративным документам.
* **PostgreSQL:** Хранит демо-данные (таблицы сотрудников, продажи).

### 2.6. Monitoring Engine (Телеметрия)
* **Prometheus + cAdvisor:** Автоматически собирают аппаратные метрики со всех Docker-контейнеров.
* **Langfuse Logger:** Записывает каждый шаг: сам запрос, ответ модели, сработавшие фильтры безопасности и затраченное время.

## 3. Распределение ролей и обязанности

### 3.1. Роль 1: Тимлид + AI-Архитектор
* **Зона ответственности:** API Layer (FastAPI).
* **Обязанности:** Создает непреодолимый барьер между интерфейсом и ИИ. Пишет логику роутера, внедряет фильтры `GuardIn` и `GuardOut`, настраивает отправку логов в Langfuse.
* **Взаимодействие:** Принимает от AI-разработчика модули (агентов, RAG) через Pull Requests и "прячет" их за роутером. Передает Red Team готовые эндпоинты для атак.

### 3.2. Роль 2: AI-разработчик
* **Зона ответственности:** AI/LLM Layer, Data Layer, Interface Layer.
* **Обязанности:** Пишет пайплайны генерации, настраивает ChromaDB, программирует логику LangGraph-агента и уязвимого MCP Server. Создает фронтенд на Streamlit.
* **Взаимодействие:** Берет PDF-документы у Target Analyst для RAG. Оформляет логику в виде изолированных функций и передает Тимлиду для встраивания в API.

### 3.3. Роль 3: DevOps-инженер
* **Зона ответственности:** Инфраструктура и Monitoring Engine.
* **Обязанности:** Настраивает Ubuntu, Docker-сети, пробрасывает GPU. Пишет `docker-compose.yml` и конфигурации Prometheus.
* **Взаимодействие:** Разворачивает инфраструктуру для Тимлида. Готовит изолированную среду Kali для Red Team.

### 3.4. Роль 4: Аналитик ИБ (Red Team Lead)
* **Зона ответственности:** Внешний контур (Атаки).
* **Обязанности:** Атакует систему через Garak и кастомные скрипты. Пытается обойти `GuardIn` прямыми инъекциями и заставить Агента выполнить вредоносный код (RCE).
* **Взаимодействие:** Начинает работу после отмашки Тимлида об этапе готовности API. Отправляет баг-репорты об успешных обходах.

### 3.5. Роль 5: Аналитик ИБ (Target и Threat Intelligence)
* **Зона ответственности:** Подготовка "мишеней" и аналитика угроз.
* **Обязанности:** Создает зараженные PDF-документы (со скрытыми инструкциями) для RAG, составляет карты уязвимостей и сценарии атак.
* **Взаимодействие:** Передает PDF-файлы AI-разработчику для векторизации. Снабжает Red Team векторами для атак.

### 3.6. Роль 6: Техписатель + QA
* **Зона ответственности:** Документация и End-to-End тестирование.
* **Обязанности:** Проверяет систему от лица пользователя. Пишет отчеты (REPORT.md, README.md), записывает видеодемонстрацию работы Guardrails.
* **Взаимодействие:** Собирает данные со всех участников. Формирует финальный пакет для куратора.

## 4. Workflow: Как команда работает вместе

Мы используем методологию Feature Branch Workflow.

### 4.1. Ветки Git
* `main`: Стабильная версия архитектуры. Защищена от прямых пушей.
* `develop`: Главная ветка интеграции. Сюда сливаются все готовые фичи.
* `feature/*`: Ветки для разработки конкретных задач (например, `feature/api-guardrails` или `feature/rag`).

### 4.2. Правила слияния (Merge Rules)
1.  **Создание ветки:** Каждая новая задача выполняется в отдельной ветке, созданной от `develop`.
2.  **Коммиты:** Используйте семантические теги: `feat:`, `fix:`, `docs:`, `infra:`, `attack:`.
3.  **Pull Request (PR):** После завершения фичи создается PR в `develop` с подробным описанием интерфейса (что принимает на вход, что отдает).
4.  **Code Review:** Тимлид (или другой участник) обязан проверить код на безопасность и логику перед слиянием (Merge).
5.  **Релиз:** В `main` ветка `develop` сливается только в конце спринта после полного QA-тестирования.

### 4.3. Пример взаимодействия (Handoff)
* **AI-dev -> Тимлиду:** AI-разработчик завершает логику Агента, пушит в `feature/agent` и открывает PR. В описании пишет: *"Агент готов. Функция `run_agent(query)` принимает строку, возвращает строку. Зависит от MCP."* Тимлид проводит ревью, сливает в `develop` и встраивает `run_agent` за роутером в `main.py`.
* **Target Analyst -> AI-dev:** Аналитик кладет новые отравленные PDF в папку `target_docs` в ветке `feature/target-mockup`. После мержа, AI-разработчик локально запускает скрипт векторизации базы.
* **Тимлид -> Red Team:** Тимлид деплоит `develop` и пишет в чат: *"Система обновлена. GuardIn и GuardOut активны. Эндпоинты открыты, атакуйте"*.

## 5. Пример возможной структуры репозитория

Привожу вариант просто для примера, чтобы было от чего отталкиваться.

```text
/smart-wiki-cyber-range
├── .env.example                  # Шаблон переменных окружения (API keys, DB ports)
├── Makefile                      # Команды: make deploy, make test, make ingest
├── infra/                        # [DEVOPS] Инфраструктура
│   ├── docker-compose.yml        # Изоляция сетей, БД, Ollama, API
│   ├── install_ubuntu.sh         # Настройка хоста и GPU
│   ├── install_kali.sh           # Настройка инструментов атакующего
│   └── prometheus/               # Конфиги сбора метрик
├── src/                          # Исходный код серверов
│   ├── api/                      # [ТИМЛИД] API Layer (FastAPI)
│   │   ├── main.py               # Роутер и эндпоинты
│   │   ├── guard_in.py           # Входная фильтрация
│   │   ├── guard_out.py          # Выходная цензура
│   │   └── langfuse_logger.py    # Отправка трейсов в Langfuse

│   │   └── mcp_server/           # Уязвимые инструменты (read_file, sql_run)

│   └── scenarios.md              # Карта векторов атак
├── red_team/                     # [RED TEAM] Атаки и эксплойты
│   ├── scripts/                  # Скрипты: attack_xss.py, attack_rce.py
│   ├── payloads/                 # Словари инъекций в JSON
│   └── tools_config/             # Конфигурации Garak, Promptfoo
└── docs/                         # [ТЕХПИСАТЕЛЬ] Документация
    ├── README.md                 # Инструкция по развертыванию
    ├── ARCHITECTURE.md           # Mermaid-схемы и описание слоев
    └── REPORT.md                 # Итоговый отчет по инцидентам
